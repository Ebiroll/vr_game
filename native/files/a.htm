              • Lennart Grahl • 7 months ago

                Great detailed explanation and thanks for the benchmark!

                While the RTT is better, bear in mind that throughput for data channels is only decent and message sizes are limited in both Mozilla's and Google's implementations at the moment (see this article for details on performance and this article for an explanation why this limitation exists). To send larger messages, you will need to fragment and reassemble messages. For example by using the SaltyRTC binary chunking implementation.

                Furthermore, I guess that a massive amount of clients to your game server will also require a massive amount of threads due to the threaded nature of the library.

                So, if at one point you feel limited by any of these issues or you're just curious, take a look at our (currently a bit feature-lacking) data channel implementation in C called RAWRTC which addresses these issues.
                For browser interop, you will need to provide an ORTC-to-WebRTC shim in the browser but this limitation will fade once we add the WebRTC API to our implementation.

                  • Brian Ho Mod Lennart Grahl • 7 months ago

                    Thanks for the feedback! As for the issues you pointed out, I'm fortunately using WebRTC data channels in a real-time gaming context, so the limiting factor is going to be the connection. At 20 updates/second in both directions, even messages over 2kb might be pushing the upper limits of some players' connections.

                    In addition, I'm designing my game around a maximum of 20 or so concurrent connections per server which in my (admittedly basic) testing, seems perfectly fine for a single WebRTC signaling thread. In the event I need to scale up to allow for more players in a given region, spinning up a few extra threads doesn't sound terrible if each one can handle 20 connections. Even in the worst case where I have more threads than processors, I'm hopeful that the I/O heavy nature of the signaling threads will lend itself to better thread scheduling.

                    Your points are definitely valid, and I'm just lucky that they don't matter that much for my specific use case. Regardless, RAWRTC looks super promising, and a more lightweight solution than including the entire Chromium WebRTC library is very welcome. I'm not terribly keen on context switching back into my game's networking framework just yet, but I'll be sure to play around with RAWRTC in the future!
